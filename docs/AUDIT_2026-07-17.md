# Registre des constats
## Fail-closed & invariants (FC)

_This codebase takes its fail-closed promise unusually seriously: the validation triple is re-derived at submit time in two independent places, the write layer is structurally separated from the read layer, the click path is guarded by validity/obstruction/dropdown probes that each refuse on uncertainty, and secrets are handled with genuine defense-in-depth. The two real gaps are both on the success-proof side rather than the write-gate side: the post-save 'gone from feed' verification treats an unreadable feed (mid-run login bounce, silent CDP timeout, or a feed longer than max_pages) as proof of disappearance — inferring success from absence of data rather than a positive scan-completed signal, and making that false CREATED sticky in the admin history — and the offer[targets][] Enter fallback can trigger an implicit form submission before the validity gate and outside the official Create click. The remaining findings (AKS_TARGET override, per-process guard state, id-existence-as-validation, declared-not-verified R24 mode) are mostly documented trade-offs where the stated invariant is stronger than the code's enforcement; they are honest, but they mean several 'hard' gates are conventions backed by operator discipline rather than mechanism.</dimension_summary>
</invoke>_

### FC1 · HIGH · Post-save 'gone' proof is fail-open when the feed is unreadable: a mid-run login bounce or CDP timeout is counted as a verified creation

- **Statut : FIXED (2026-07-17)** — post-save désormais bâti sur un walk positivement complet : CdpCommandError/NotLoggedInError/FeedScanError remontent, l'offre courante passe UNKNOWN et le run stoppe (`stopped="feed_unreadable"`)
- **Vérification :** confirmé
- **Fichier :** `src/submitter.py:280`
- **Dimension :** fail-closed

_verify_gone() concludes gone=True whenever the stop_on scan ends without finding the offer. But _scan_feed() ends the scan on the FIRST page whose rows come back empty ('if not rows: break'), and rows come back empty not only on a genuinely empty feed page but also when (a) the WP session expired mid-run and the feed URL bounced to wp-login (no tr[data-offer] nodes; is_login_page() is checked only in the pre-flight of run() and in fetch_session_catalog, never inside _scan_feed), or (b) the CDP command silently timed out — cdp_session._cmd() returns {"error": "timeout"} instead of raising, _evaluate() extracts value=None from it without error, and page_offer_rows() maps that to []. In both cases the offer is declared 'gone from feed', the entry gets submitted=true/success=true, and offer_submit_history() in src/admin/submit_manager.py makes 'created' STICKY ('a later not-in-feed failure never demotes it') and start_submit() refuses ever re-approving that offer ('ré-ajout interdit'). So a transient socket hiccup during the verify walk converts an unproven (possibly failed) write into a permanent, unfalsifiable CREATED record, and the offer can silently never be entered. The initial-index direction of the same weakness fails closed (empty index → 'not in current feed' blockers), so this is specifically the success-proof direction that is fail-open. This contradicts the core promise that success comes only from a positive deterministic signal — here success is inferred from ABSENCE of data, with no distinction between 'scanned the feed and it is gone' and 'could not scan the feed'.

> Preuve : src/submitter.py:279-282: 'self.session.navigate(url); rows = self.session.page_offer_rows(); if not rows: break' → _verify_gone (446-451): 'index, by_url, found = self._scan_feed(...stop_on=offer_id...); if found: return False, None, None; return True, index, by_url'. src/cdp_session.py:217: 'return {"error": "timeout", "method": method}' (no raise), and _evaluate (119): 'return result.get("result", {}).get("value")' → None → page_offer_rows returns []. submit_manager.py:127-129: '``created`` (confirmed added on AKS — sticky, a later "not in feed" failure never demotes it)'.

### FC2 · MEDIUM · AKS_TARGET=vps environment variable forces authoritative=true anywhere, making the 'read-only until green on the Debian VPS' gate spoofable by one env var

- **Statut : FIXED (2026-07-17)** — force AKS_TARGET=vps supprimée ; autorité = marqueur root /etc/aks-executor.target (possédé root, non world-writable, contenu == hostname) ; shell audit aligné ; gate vérifié vert live
- **Vérification :** confirmé
- **Fichier :** `src/aks_env.py:222`
- **Dimension :** fail-closed

Every write gate (scripts/05_submit.py, scripts/00b_login.py, scripts/02_extract_feed.py) keys on report['authoritative'], which comes from current_environment(). That function honors an AKS_TARGET=vps override that unconditionally sets the target marker on — on any Linux box (classify_environment only additionally requires system=='Linux'). CLAUDE.md states the hard constraint 'Do not try to force a local green result by changing environment variables' and INVARIANTS docs say authoritative is true ONLY on the Debian VPS target, but the code itself supplies exactly the environment-variable lever the constraint forbids. The override is documented (docs/CONTRIBUTING.md, and docs/NOOB.md warns against misuse) and exists as a test seam, yet nothing technical restricts it to tests: a sandboxed CI Linux runner with AKS_TARGET=vps and a reachable CDP endpoint would pass the write gate. The gate is therefore a convention, not an enforcement, off-target.

> Preuve : src/aks_env.py:222-228: 'override = os.environ.get("AKS_TARGET", "").strip().lower(); if override == "vps": marker = True ... else: marker = os.path.exists(TARGET_MARKER_PATH)' feeding classify_environment(... target_marker_present=marker ...) → 'is_target = system == "Linux" and target_marker_present; ... "authoritative": is_target'. CLAUDE.md: 'Do not try to force a local green result by changing environment variables'.

### FC3 · MEDIUM · StepGuard blocks are per-process and in-memory only — a hard block does not survive re-running the script, even with the same run_id

- **Statut : FIXED (2026-07-17)** — BlockLedger (runs/<id>/guard_ledger.json) : G03 inter-processus — 1 passe de récupération libre, 2 passes bloquées consécutives → 3e exige --acknowledge-block
- **Vérification :** confirmé
- **Fichier :** `src/step_guard.py:136`
- **Dimension :** fail-closed

The guard's whole promise ('a block ... stays blocked until a *new* task is started', 'a task can never clear its own block') holds only within one Python process: all state lives in instance attributes, snapshot() is written to logs but never read back, and every CLI invocation constructs a fresh guard (e.g. _SubmitterBase.__init__ builds its own default guard; 05_submit.py never rehydrates a prior block). So a run that hard-blocked on ten_consecutive_failures can be immediately re-launched with the identical run_id/approved.json and the guard starts clean — start_task(run_id) on a fresh object is a 'new task' by construction even though the intent is the same. The process-level compensations (exit code + submit_plan.json must be read before continuation, per CLAUDE.md; the admin SubmitManager's one-at-a-time lock and already-created gate) reduce the practical exposure, but the documented property that a block is 'un-negotiable' is stronger than what the code enforces; nothing distinguishes 'genuinely new task' from 'same task, new process'.

> Preuve : src/step_guard.py:136-145: '_task_id/_blocked/_history' are plain instance attributes; no persistence path exists (snapshot() at 329 is write-only into RunLogger). src/submitter.py:199-204: 'self.guard = guard or StepGuard(...)' — new per construction; scripts/05_submit.py:291 constructs a new submitter per invocation; run() line 492: 'self.guard.start_task(run_id)' always succeeds on the fresh guard.

### FC4 · MEDIUM · resolve_catalog_id id-fallback validates existence, not meaning: the candidate's label is never compared to the catalog text on the id path

- **Statut : FIXED (2026-07-17)** — chemin id : le label doit apparaître en mot entier dans le texte du catalogue live — un id dérivé vers une autre option → None → blocker
- **Vérification :** confirmé
- **Fichier :** `src/submitter.py:124`
- **Dimension :** fail-closed

The write path re-resolves region/edition against the live session catalog precisely because 'ids drift; the matcher's are not authoritative'. Path 1 (unique label match) is sound. But path 2 accepts the matcher's hardcoded id (src/matcher.py REGION_IDS is a static table) merely because that key EXISTS in the live catalog, adopting the catalog's canonical text without ever checking that this text is compatible with the candidate's label. Region labels ('EU') structurally never match the composite catalog texts ('Steam EU (9)' normalizes to 'steam eu'), so regions routinely take the id path — meaning the region actually entered is 'whatever id 9 denotes in today's catalog', flagged changed:False. If AKS ever reassigns an id (the documented failure mode that motivated the whole catalog mechanism — 'ids change as AKS adds editions/regions'), this path writes the wrong region/edition with no blocker and no CHANGED flag in the report. A cheap fail-closed tightening exists: require the candidate label's tokens to appear in the catalog option's text before trusting the id.

> Preuve : src/submitter.py:124-131: 'by_id = {str(o.get("key")): o for o in master_options}; if str(candidate_id) in by_id: o = by_id[str(candidate_id)]; return {"id": str(candidate_id), "text": o.get("text"), "source": "id", ..., "changed": False}' — no comparison between o['text'] and the candidate's label. Docstring at 96-98 concedes: 'The dropdowns are a global catalog whose ids drift ... so the matcher's hardcoded id is not authoritative'.

## Matcher (MA)

_The matcher is an unusually well-documented, well-tested rule pile whose fail-closed philosophy genuinely holds on most paths — every past escape has a coded rule, a doc section, and a named regression test, which is rare discipline. However, verified live repros show the remaining holes cluster exactly where the module deviates from its own conventions: substring matching where everything else is word-boundary (explicit_platform turning 'Epic Chef … Steam Key' into an EPIC candidate, '-gift' matching 'gifted'), noise-whitelisting without a qualifier backstop (Skyrim 'Anniversary Edition' verified entering the base page as Standard), transient-error shadowing in resolve_aks that contradicts its own docstring, and three documented §4.3/§4.4 skips (EU-NA, country gifts, Gamivo -en-) that were never coded. The rule-pile architecture is near its sustainability limit — match_offer's 200-line body with R18/R20/R23/R25/R27 interleaved, two inconsistent editions-shape handlers (one of which crashes), and asymmetric fail-open/fail-closed behavior on markup drift all argue for the half-emerged table-driven layer (classification tuples, REGION_IDS, EDITION_HINTS) to be completed into a uniform boundary-matched, page-verified pipeline before the next ten rules land._

## Submitter & couche CDP (SC)

_The submitter's write path is impressively defended — every past incident (wrong-edition force, open-dropdown overlay, invalid-form silent click) has a specific deterministic guard, all JS interpolation is injection-safe via json.dumps, and post-save disappearance under dual identity is the sole success proof. ## Admin web (AS)

_The admin web app is unusually well-defended for a stdlib-only internal tool: CSRF defense (custom header + content-type + Origin/Host, no CORS), genuinely closed path traversal (regex + resolve + parent check + filename whitelist), argv-list subprocess spawning with server-side re-validation of mode/canary/triple, and every real-write gate re-enforced in the child script itself (invariants green+authoritative, triple re-derivation, typed-GO required server-side).
